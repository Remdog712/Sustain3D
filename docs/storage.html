Baseline storage model (small institution)

The goal is to prevent silent loss: drive failure, accidental deletion, and “bit rot.”

Minimum: 3 copies

  • Working copy (editable): where staff process files.
  • Local backup: separate drive/system, not the same device.
  • Offsite copy: cloud or a physically separate location.
Duke Libraries policy example:
“Fixity checks are performed… at regular intervals.”

If you can’t do regular fixity, you don’t know if files are silently corrupting.

Fixity (checksums) in plain language

A checksum is a fingerprint of a file. If the fingerprint changes, the file changed or was corrupted.

Routine (recommended)

  1. Create checksums for 03_preservation/ (SHA-256 recommended).
  2. Store the checksum manifest in Preservation_Logs/checksums/.
  3. Re-run checks on a schedule (quarterly is a realistic start).
DPC Handbook tip:
“Many of the benefits of fixity checking… only… if there are multiple copies…”

Fixity without backup copies just tells you something broke—you still can’t repair it.

Storage tiers (to keep things manageable)

  • Working: active project files and edits (changeable).
  • Preservation: masters (locked / no edits).
  • Access: web/teaching derivatives (replaceable).

Good / Better / Best

Good

  • 2 copies total (working + backup).
  • Manual monthly spot checks.

Better

  • 3 copies (includes offsite).
  • Checksums for preservation masters.

Best

  • Automated fixity schedule.
  • Documented disaster recovery plan.

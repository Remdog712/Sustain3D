Goal

Use written standards and policy baselines so naming, access, rights, and publication decisions stay consistent from capture through preservation and release.

1) Workflow standards document (author then follow)

  • File: Documentation/workflow_standards.txt
  • Version: set by your team (for example 1.0)
  • Date: use your current publication date for the standard.
  • Owner: person, team, or organization responsible for maintenance.
  • This document should be written by the organization or project lead, then used as the mandatory workflow rule set for all objects in the project.

2) Rules enforced by workflow_standards.txt

  • Character rules: use only A-Z, a-z, 0-9, underscore; no spaces, hyphens, or special characters.
  • Object naming: <Collection>_<4digitID>_<ShortLabel> with permanent zero-padded ID.
  • Workspace policy: Projects folder is temporary processing space; Objects/Preservation is long-term storage.
  • Preservation masters: <ObjectID>_<asset>_v##_pres.<ext>; never overwrite old versions.
  • Access derivatives: optimized files with separate naming and no replacement of preservation masters.
  • Print derivatives: manifold repair allowed, but document edits in paradata.
  • Do not: use ambiguous filenames (for example final.obj) or rename objects after DOI assignment.

3) Generic policy values recorded in project_registry.json

"policy": {
  "access_level": "",
  "rights": "",
  "license": "",
  "attribution": ""
},
"publication": {
  "doi_status": "none",
  "doi_value": null,
  "landing_page": null
}

4) Final review checks before release

  1. Confirm object filenames and folder names match Documentation/workflow_standards.txt.
  2. Confirm preservation packages are in repository preservation storage and not only in project working folders.
  3. Confirm policy values in project_registry.json match object metadata rights and access fields.
  4. Confirm DOI status and publication landing page fields are current before public release.

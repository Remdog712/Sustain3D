Goal

Set a file and folder structure that stays understandable when one machine has multiple parts, unknown components, and assembly relationships.

Use these patterns so parts remain independently preservable while the full assembly stays reconstructable.

1) Choose the structural model

You have two valid structural models:

Object-centric

Best when the machine itself is the long-term intellectual object you care about.

Project-centric

Best for documenting workflow context for one campaign, survey, or time-bounded initiative.

For complex machines, use both: project structure for workflow and object structure for durable identity.

2) Recommended structure: object with parts subfolder

If the machine is the intellectual "thing" you care about long-term, keep it as a parent object with child parts.

Objects/
  Mechanical/
    WaterPump_001/
      assembly/
        WaterPump_001_master_assembly_v01.pres.obj
        assembly_metadata.txt
      parts/
        WP001_P01_Impeller/
        WP001_P02_Housing/
        WP001_P03_Shaft/
      documentation/
        metadata.txt
        paradata.txt
        rights_and_access.txt

Each part uses the same internal structure

WP001_P01_Impeller/
  01_raw/
  02_processing/
  03_preservation/
  04_access/
  documentation/

Why this works

  • The machine is a parent object.
  • Parts are child objects.
  • Each part remains independently preservable.
  • The assembly model defines how parts fit together.

3) Naming convention for parts

Use a consistent part code:

WaterPump_001_P01_Impeller
WaterPump_001_P02_Housing

Pattern: [ParentID]_P[##]_[PartName]

  • If order matters physically, use sequential numbering.
  • If order does not matter, use functional labels.

4) What if you do not know the name?

Never invent a false authoritative name. Use a placeholder you can defend.

Option A - Descriptive placeholder

InsectSpecimen_045_unknown_wing_fragment
Machine_002_unknown_component_A

Option B - Functional description

Machine_002_rotating_disc_component
Artifact_014_small_curved_metal_piece

Option C - Controlled placeholder

UnknownObject_001
Unidentified_Component_003

Then document uncertainty in metadata:

identified_as: unknown
suspected_function: ventilation coupling
confidence_level: low
identified_by: J. Orange
date_identified: 2026-02-20

Key rule: file names describe what you know. Metadata describes uncertainty.

5) Representing how parts fit together (critical)

File structure alone is not enough. You need structural hierarchy, assembly metadata, and relationship documentation.

A. Assembly folder in parent object

assembly/
  WaterPump_001_master_assembly_v01.pres.obj
  assembly_relationships.json

B. Relationship metadata (recommended)

{
  "parent_object": "WaterPump_001",
  "parts": [
    {
      "id": "WP001_P01_Impeller",
      "connects_to": "WP001_P03_Shaft",
      "connection_type": "rotational_fit",
      "alignment_reference": "central_axis"
    },
    {
      "id": "WP001_P02_Housing",
      "contains": "WP001_P01_Impeller",
      "connection_type": "enclosure"
    }
  ]
}

C. Cross-reference inside each part metadata

parent_object: WaterPump_001
part_number: P01
assembly_position: internal rotating component
fits_with: WP001_P03_Shaft

This creates bidirectional traceability between parent assembly records and child part records.

6) When to use project vs object structure

If the machine is scanned within a larger survey, keep both contexts:

Projects/
  FactorySurvey_2026/
    Machine_12_WaterPump/

Objects/
  Mechanical/
    WaterPump_001/
  • Projects document workflow context and campaign activity.
  • Objects document long-term intellectual identity.

Do not collapse these into one folder tree. That usually breaks long-term usability.

7) Clean rules summary

  1. Every physical thing gets one object folder.
  2. Every removable or distinct component gets a child object folder.
  3. Assemblies define relationships, not filenames.
  4. Unknown components use placeholders plus documented uncertainty.
  5. Projects record workflow; objects record long-term identity.

Copy-ready templates for multi-part machines

Use these commands to scaffold parent object, part folders, and relationship files quickly.

Create parent object with part folders

$base = ".\3D_Repository\Objects\Mechanical\WaterPump_001"
New-Item -ItemType Directory -Path @(
  $base,
  (Join-Path $base "assembly"),
  (Join-Path $base "parts\WP001_P01_Impeller"),
  (Join-Path $base "parts\WP001_P02_Housing"),
  (Join-Path $base "parts\WP001_P03_Shaft"),
  (Join-Path $base "documentation")
) -Force | Out-Null

New-Item -ItemType File -Path @(
  (Join-Path $base "assembly\assembly_relationships.json"),
  (Join-Path $base "documentation\metadata.txt"),
  (Join-Path $base "documentation\paradata.txt"),
  (Join-Path $base "documentation\rights_and_access.txt")
) -Force | Out-Null

Create standard internal folders for one part

$part = ".\3D_Repository\Objects\Mechanical\WaterPump_001\parts\WP001_P01_Impeller"
New-Item -ItemType Directory -Path @(
  (Join-Path $part "01_raw"),
  (Join-Path $part "02_processing"),
  (Join-Path $part "03_preservation"),
  (Join-Path $part "04_access"),
  (Join-Path $part "documentation")
) -Force | Out-Null

@"
parent_object: WaterPump_001
part_number: P01
assembly_position:
fits_with:
identified_as:
suspected_function:
confidence_level:
identified_by:
date_identified:
"@ | Set-Content (Join-Path $part "documentation\metadata.txt")

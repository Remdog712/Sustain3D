Goal

Use predictable names and folder patterns so 3D packages stay connected and understandable.

Apply one naming standard across scan data, derivatives, and 3D print outputs.

Recommended naming pattern

[ObjectOrSite]_[MethodOrProcess]_[YYYYMMDD]_v[##]_[stage].[ext]

What each part means

  • [ObjectOrSite] stable ID for the scanned object or site.
  • [MethodOrProcess] capture or processing method (photogrammetry, lidar, mesh, printprep).
  • [YYYYMMDD] date in sortable format.
  • v[##] version number such as v01, v02, v03.
  • [stage] raw, proc, pres, access, prn.
  • [ext] file extension for the specific output format.

Examples

TempleA_LiDAR_20240215_v01_raw.e57
TempleA_mesh_clean_20240220_v03_proc.obj
TempleA_web_20240220_v01_access.glb
TempleA_printprep_20240222_v02_prn.3mf

3D file formats: what is better for what

Keep at least one long-term preservation format and one web-access format for each final model.

Preservation formats (long-term copies)

  • OBJ + MTL + texture files: Open and widely supported. Good for textured meshes and long-term reuse across tools.
  • PLY: Good for high-detail geometry and vertex color. Common in research workflows and easy to validate.
  • E57: Strong choice for point clouds from LiDAR or structured light. Preserves scan context better than mesh-only exports.
  • STL: Useful mainly for 3D printing geometry. Not ideal as the only preservation format because it usually lacks color, texture, and rich metadata.

Access formats (web and sharing copies)

  • GLB (binary glTF): Best default for web viewers. Single file, fast loading, and supports modern materials.
  • glTF + external textures: Better when you need editable web assets and separate texture management.
  • USDZ: Useful for Apple AR experiences. Keep it as an access derivative, not the only archival master.
  • 3MF: Better than STL for sharing print jobs because it can include units, color, and print settings.

Practical rule: preserve rich, open source files in 03_preservation, publish lightweight delivery files in 04_access.

Versioning and linked files

  • Keep source files unchanged after capture or ingest.
  • Save milestone versions during processing rather than overwriting.
  • Store linked files (geometry, materials, textures, print profiles) in the same package folder.
  • Open outputs on another machine to validate that links still work.

Optional terminal templates

These can be pasted into PowerShell to create folder scaffolding quickly.

Repo-level directory template

$repo = ".\3D_Repository"
New-Item -ItemType Directory -Path $repo,(Join-Path $repo "Projects"),(Join-Path $repo "Objects"),(Join-Path $repo "Metadata"),(Join-Path $repo "Documentation"),(Join-Path $repo "Preservation_Logs") -Force | Out-Null
New-Item -ItemType File -Path (Join-Path $repo "Metadata\master_inventory.csv") -Force | Out-Null
New-Item -ItemType File -Path (Join-Path $repo "Metadata\project_registry.json") -Force | Out-Null

Project-level directory template

$project = ".\3D_Repository\Projects\ProjectName_ObjectID"
New-Item -ItemType Directory -Path $project,(Join-Path $project "01_raw_scans"),(Join-Path $project "02_processing"),(Join-Path $project "03_preservation"),(Join-Path $project "04_access"),(Join-Path $project "documentation"),(Join-Path $project "print") -Force | Out-Null
New-Item -ItemType File -Path (Join-Path $project "documentation\metadata.txt"),(Join-Path $project "documentation\paradata.txt"),(Join-Path $project "documentation\rights_and_access.txt"),(Join-Path $project "print\printer_notes.txt") -Force | Out-Null

Why file management matters

A 3D “model” is usually a package: geometry + textures + material files + documentation. If those links break, you can end up with a file that opens but is incomplete or misleading.

1) Naming conventions (simple and durable)

Institutional tip (Harvard HMS):
“File names should be 40-50 characters…”

Short names reduce failures across systems and long file paths.

Recommended pattern

[ObjectID]_[Method]_[YYYYMMDD]_v[##]_[stage].[ext]
Example:
PAOAN_LiDAR_20240215_v01_raw.e57
PAOAN_mesh_20240220_v03_pres.obj
PAOAN_web_20240220_v01_access.glb

Rules

  • Use only letters/numbers, underscores, and dashes.
  • Put the version at the end of the “core name” (v01, v02, v03).
  • Use ISO-style dates (YYYYMMDD) so sorting is consistent.

2) Versioning (what to keep)

Keep versions at major workflow milestones. Do not overwrite.

  • Raw: original photos/scans/point clouds (never overwrite).
  • Working: project files (Metashape project, Blender files, etc.).
  • Preservation master: highest-fidelity export you intend to preserve long-term.
  • Access derivatives: GLB/USDZ/decimated meshes for web/teaching.
Workflow tip (DCN CURATE(D)):
“Transform file formats for reuse…”

Preserve the master first; transform for access second.

3) Keep linked files together (avoid broken assets)

Many 3D formats reference external files. Example: an OBJ often relies on a separate MTL file and multiple textures.

Packaging rules

  • Store geometry + materials + textures in the same folder.
  • Use relative paths (not absolute paths) when exporting.
  • After export, validate by opening the package in a different viewer/computer.
Format sustainability idea (Library of Congress):
“Seven factors… influence the feasibility and cost of preserving content…”

File formats are evaluated partly by transparency, documentation, and dependencies.

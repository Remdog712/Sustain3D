Goal

Create a stable structure that separates repository-wide program management from project-by-project working folders, while keeping naming, documentation, and long-term stewardship consistent.

Repository level vs project level

Repository-level structure stores shared program components such as object groupings, metadata standards, and governance documentation across many projects. Project-level structure stores one project package from raw capture through access outputs. Together they keep institution-wide continuity while preserving clear project context.

Purpose of each repository folder

  • Documentation/ local standards and governance files, including workflow_standards.txt.
  • Metadata/ project-level registry records, including project_registry.json.
  • Objects/ durable object identities and optional long-term object groupings.
  • Preservation/ long-term stored package for each object: raw package, preservation master package, and metadata file.
  • Preservation_Logs/ operational log files for migrations and preservation actions over time.
  • Projects/ temporary processing workspace used to move from raw scans to preservation and access outputs.

2) Project workspace template (empty)

Use this folder structure for each object while processing. Final packages are then copied into Preservation/.

Projects/
  [Project_ID]/
    [Object_ID]/
      01_raw_scans/
      02_processing/
      03_preservation/
      04_access/
      documentation/

Purpose of each project workspace folder

  • 01_raw_scans/ raw capture imports and original source files.
  • 02_processing/ active editing files and intermediate outputs.
  • 03_preservation/ finalized high-fidelity preservation exports prepared for handoff.
  • 04_access/ optimized derivatives for web, teaching, or outreach.
  • documentation/ object metadata and paradata notes created during processing.

Preservation handoff template

Preservation/
  [Project_ID]/
    [Object_ID]/
      [Object_ID]__v##_raw/
      [Object_ID]_v##_pres/
      [Object_ID]_Metadata.txt

3) Rules

  • Every object keeps one stable object ID across raw, processing, preservation, and access files.
  • Projects are working space; Preservation is long-term retention space.
  • Raw package, preservation package, and object metadata must be copied into Preservation at handoff.
  • Project registry and workflow standards are maintained at repository level for consistency.

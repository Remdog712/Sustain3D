Goal

Create a stable repository structure that can hold many projects over time.

Where this fits

This step defines structure before detailed file naming, metadata entry, storage decisions, and policy publishing.

Repository-level structure (all projects)

The repository-level structure is the organization for your full 3D program. It is not one project, it is the container that holds all projects, shared metadata, shared policies, and shared references. 

3D_Repository/
  Projects/
  Objects/
  Metadata/
    master_inventory.csv
  Policies/
  Resources/

Purpose of each repository folder

  • Projects/ stores each project working package by project ID.
  • Objects/ can store object-centric groupings across multiple projects.
  • Metadata/ stores cross-project inventory and lookup files.
  • Policies/ stores rights and access templates used by all projects.
  • Resources/ stores internal references, guides, and implementation notes.

Project-level structure (one project)

The project-level structure is for one specific scan effort. Every project should use the same structure so staff can move between projects without relearning folder logic. 

ProjectName_ObjectID/
  01_raw/
  02_processing/
  03_preservation/
  04_access/
  metadata.txt
  paradata.txt
  policy.txt

Purpose of each project folder

  • 01_raw/ source captures exactly as collected.
  • 02_processing/ in-progress project files and intermediate outputs.
  • 03_preservation/ finalized preservation master files.
  • 04_access/ delivery copies for web, teaching, and outreach.

Rules

  • Every project must have one stable object/project ID.
  • Preservation files stay separate from access copies.
  • Metadata, paradata, and policy files stay in the project root so they are always visible.
  • Repository inventory tracks DOI status for each project.

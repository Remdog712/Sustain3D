Goal

Create a stable repository structure that can hold many projects over time.

Design once, then reuse the same pattern across heritage scans, born-digital models, and 3D print files.

Purpose of each repository folder

  • Projects/ stores each initiative as a complete working package.
  • Objects/ groups reusable outputs by type (including print-oriented objects).
  • Metadata/ stores cross-project inventory and registry records.
  • Documentation/ stores local standards so teams apply the same rules each cycle.
  • Preservation_Logs/ stores fixity and migration history for long-term tracking.

Project-level hierarchy (scan or print focused)

Use one repeatable hierarchy for every project so lifecycle stages remain clear and files stay traceable.

Purpose of each project folder

  • 01_raw_scans/ raw captures or original downloaded files.
  • 02_processing/ active edits and intermediate processing outputs.
  • 03_preservation/ final preservation masters retained without overwrite.
  • 04_access/ classroom, web, and outreach derivatives.
  • print/ 3MF/STL/G-code outputs plus print process notes.

Rules

  • Every project package uses one stable ID that is reused across scan, preservation, access, and print files.
  • Preservation files always stay separate from access and print-ready outputs.
  • Documentation files are required and updated at each major step.
  • Repository inventory tracks status, storage location, and publication readiness.
